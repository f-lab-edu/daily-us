# 1. 인덱스 설계

`docs/01_Requirements/기능_정의_02.md`의 기능 요구사항과 ERD 기준 테이블명을 바탕으로, `SELECT` 기준 주요 조회 패턴만 인덱스 관점에서 정리한다.

## 1-1. 조회 조건
| 기능 | 테이블명 | 조회 조건 | 예상사용빈도 |
|---|---|---|---|
| 로그인 | `users` | `where email = ? and deleted_at is null` | 상 |
| 본인 정보 조회 | `users` | `where user_id = ? and deleted_at is null` | 중 |
| 다른 사용자 프로필 조회 | `users` | `where user_id = ? and deleted_at is null` | 중 |
| 팔로워 수 조회 | `user_follow` | `where followee = ?` | 중 |
| 팔로잉 수 조회 | `user_follow` | `where follower = ?` | 중 |
| 팔로우 여부 조회 | `user_follow` | `where follower = ? and followee = ?` | 중 |
| 본인 게시물 조회 | `posts` | `where user_id = ? and created_at >= ?` | 상 |
| 본인 월별 활동 내역 조회 | `posts` | `where user_id = ? and created_at >= ? and created_at < ?` | 중 |
| 게시글 목록 조회(피드) | `posts` | `where user_id = ? and created_at < ?` | 최상 |
| 게시글 상세 조회 | `posts` | `where post_id = ?` | 최상 |
| 게시글 이미지 조회 | `post_images` | `where post_id = ?` | 상 |
| 게시글 좋아요 여부 조회 | `post_likes` | `where post_id = ? and user_id = ?` | 상 |
| 게시글 좋아요 수 조회 | `post_likes` | `where post_id = ?` | 중 |
| 댓글 목록 조회 | `comments` | `where post_id = ? and created_at < ?` | 최상 |
| 답글 목록 조회 | `comments` | `where parent_id = ? and created_at < ?` | 중 |
| 댓글 상세 조회 | `comments` | `where comment_id = ?` | 중 |
| 댓글 좋아요 여부 조회 | `comment_likes` | `where comment_id = ? and user_id = ?` | 중 |
| 댓글 좋아요 수 조회 | `comment_likes` | `where comment_id = ?` | 중 |
| 본인 그룹 조회 | `group_members` | `where user_id = ?` | 중 |
| 그룹 참여 여부 조회 | `group_members` | `where group_id = ? and user_id = ?` | 중 |
| 그룹원 조회 | `group_members` | `where group_id = ?` | 중 |
| 그룹 상세 조회 | `groups` | `where group_id = ?` | 상 |
| 그룹 내 랭킹 조회 | `group_members` + `posts` | `where group_id = ? and user_id = ?, where user_id = ? and created_at >= ? and created_at < ?` | 상 |
| 해시태그 게시글 조회 | `hashtage_posts` | `where hashtag_id = ? and post_id = ?` | 하 |
| 해시태그명 조회 | `hashtag` | `where name = ?` | 하 |

## 1-2. 테이블별 PK / 인덱스 설계

| 테이블명 | PK | 인덱스 |
|---|---|---|
| `users` | `PRIMARY KEY (user_id)` | `INDEX idx_users_email_deleted_at (email, deleted_at)` |
| `user_follow` | `PRIMARY KEY (follower, followee)` | `INDEX idx_user_follow_followee (followee)` |
| `posts` | `PRIMARY KEY (post_id)` | `INDEX idx_posts_user_id_created_at (user_id, created_at)` |
| `post_images` | `PRIMARY KEY (post_image_id)` | `INDEX idx_post_images_post_id (post_id)` |
| `post_likes` | `PRIMARY KEY (post_id, user_id)` | 없음 |
| `comments` | `PRIMARY KEY (comment_id)` | `INDEX idx_comments_post_id_created_at (post_id, created_at)`, `INDEX idx_comments_parent_id_created_at (parent_id, created_at)` |
| `comment_likes` | `PRIMARY KEY (comment_id, user_id)` | 없음 |
| `group_members` | `PRIMARY KEY (group_id, user_id)` | `INDEX idx_group_members_user_id (user_id)` |
| `groups` | `PRIMARY KEY (group_id)` | 없음 |
| `hashtage_posts` | `PRIMARY KEY (hashtag_id, post_id)` | `INDEX idx_hashtage_posts_post_id_hashtag_id (post_id, hashtag_id)` |
| `hashtag` | `PRIMARY KEY (hashtag_id)` | `INDEX idx_hashtag_name (name)` |

# 2. 제약조건 및 컬럼 크기 설계

## 2-1. UNIQUE / 제약조건

| 테이블명 | 제약조건 | 설명 |
|---|---|---|
| `users` | `UNIQUE (email)` | 이메일 회원가입과 로그인 식별자. 요구사항상 중복 불가 |
| `users` | `UNIQUE (nickname)` | 프로필 식별용 닉네임. 요구사항상 중복 불가 |

## 2-2. VARCHAR / TEXT 크기 기준

| 테이블명 | 컬럼명 | 타입 | 길이/크기 기준 | 비고 |
|---|---|---|---|---|
| `users` | `email` | `VARCHAR` | `255` | 이메일 RFC 대응 여유 길이 |
| `users` | `password` | `VARCHAR` | `255` | BCrypt 등 해시 저장 고려 |
| `users` | `nickname` | `VARCHAR` | `100` | 화면 노출명 |
| `users` | `profile_image` | `VARCHAR` | `500` | CDN/스토리지 URL 저장 |
| `groups` | `name` | `VARCHAR` | `100` | 그룹명 |
| `groups` | `intro` | `VARCHAR` | `500` | 그룹 소개 |
| `groups` | `group_image` | `VARCHAR` | `500` | 그룹 이미지 URL |
| `posts` | `content` | `TEXT` | 가변 | 게시글 본문은 길이 상한이 유동적이므로 `TEXT` 사용 |
| `post_images` | `image_url` | `VARCHAR` | `500` | 이미지 URL 저장 |
| `comments` | `content` | `TEXT` | 가변 | 댓글/답글 본문 |
| `hashtag` | `name` | `VARCHAR` | `100` | 해시태그 문자열 |

# 3. 비고
- users 테이블이 소프트 삭제 방식이기 때문에 ON DELETE CASECADE/RESTRICT 대신 애플리케이션 레벨에서 제어한다.
- `users`는 현재 `deleted_at`을 사용하지만, 요구사항상 이메일/닉네임 중복은 허용하지 않는 방향으로 정리했으므로 `UNIQUE (email)`, `UNIQUE (nickname)` 기준으로 설계한다.
- 추후 탈퇴 후 재가입 정책을 허용하려면 `deleted_at`만으로는 `UNIQUE` 정책을 충분히 표현하기 어려울 수 있으므로 별도 상태 컬럼이나 generated column 전략을 검토한다.
