## 4. 공통 정책

### 4.1 인증/인가
- 본인 프로필 조회, 본인 프로필 수정, 본인 활동 내역 조회는 로그인한 사용자만 가능하다.
- 다른 사용자 프로필 조회는 공개 조회를 기본으로 가정하되 탈퇴한 사용자 정보는 반환하지 않는다.
- 팔로우/언팔로우는 로그인한 사용자만 수행할 수 있다.
- 본인 기준 기능은 액세스 토큰 사용자와 요청 대상이 동일해야 한다.

### 4.2 조회 정책
- 프로필 조회 시 비밀번호, refresh token 등 민감 정보는 반환하지 않는다.
- 프로필 조회 시 공개 가능한 범위의 회원 기본 정보와 관계 정보를 반환한다.
- 활동 내역은 월 단위로 조회한다.

### 4.3 정합성 정책
- 팔로우 관계는 `(follower, followee)` 기준으로 유일해야 한다.
- 자기 자신은 팔로우할 수 없다.
- 탈퇴 회원은 프로필 조회, 팔로우 대상, 활동 집계 대상에서 제외한다.

### 4.4 데이터 제약 조건
- 닉네임은 중복될 수 없다.
- 닉네임 최대 길이는 20자로 제한한다.
- 프로필 이미지는 1장만 설정할 수 있다.
- 자기소개 최대 길이는 160자로 제한한다.
- 팔로우는 자기 자신에게 할 수 없다.
- 팔로우는 동일 사용자 기준 중복 생성할 수 없다.
- 활동 내역 조회는 미래 날짜를 허용하지 않는다.

## 5. 기능 상세 정의

### 5.1 본인 프로필 조회

#### 목적
- 로그인한 사용자가 자신의 프로필 정보를 조회한다.

#### 입력 항목

#### 처리 규칙
- 액세스 토큰 사용자 식별자로 생성 회원을 조회한다.
- 응답에는 이메일, 닉네임, 프로필 이미지, 자기소개, 팔로워 수, 팔로잉 수를 포함한다.
- 민감 정보는 반환하지 않는다.

#### 데이터 반영
- `users`를 사용자 식별자로 조회한다.

#### 예외
- 인증 정보가 없거나 유효하지 않으면 조회할 수 없다.
- 존재하지 않거나 탈퇴한 회원이면 실패 처리한다.

### 5.2 본인 프로필 수정

#### 목적
- 로그인한 사용자가 자신의 프로필 정보를 수정한다.

#### 입력 항목
- `Authorization` access token
- 수정 대상 필드
  - `nickname`
  - `profileImage`

#### 처리 규칙
- 본인만 수정할 수 있다.
- 닉네임 변경 시 중복 검사를 수행한다.
- 프로필 이미지 URL, 자기소개 길이 등 입력 제약을 검증한다.
- 수정하지 않은 필드는 기존 값을 유지한다.
- 수정 성공 시 프로필 조회 결과에 즉시 반영되어야 한다.

#### 데이터 반영
- `users`의 수정 대상 컬럼과 `updated_at`을 갱신한다.

#### 예외
- 닉네임이 중복되면 수정할 수 없다.
- 입력값 검증 실패 시 수정할 수 없다.
- 존재하지 않거나 탈퇴한 회원이면 수정할 수 없다.

### 5.3 다른 사용자 프로필 조회

#### 목적
- 사용자가 다른 회원의 공개 프로필을 조회한다.

#### 입력 항목
- `userId`

#### 처리 규칙
- 대상 사용자가 활성 회원인 경우에만 조회할 수 있다.
- 로그인한 사용자만 가능하다.
- 공개 가능한 항목만 반환한다.
- 응답에는 닉네임, 프로필 이미지, 자기소개, 팔로워 수, 팔로잉 수를 포함한다.
- 로그인 사용자 기준으로 대상 사용자에 대한 팔로우 여부를 함께 반환할 수 있다.
- 탈퇴한 회원은 조회 대상에서 제외한다.

#### 데이터 반영
- `users`를 대상으로 회원 기본 정보를 조회한다.
- 로그인 사용자일 경우 `user_follow`를 추가 조회해 관계 여부를 계산한다.

#### 예외
- 존재하지 않거나 탈퇴한 사용자는 조회할 수 없다.

### 5.4 사용자 팔로우

#### 목적
- 로그인한 사용자가 다른 사용자를 팔로우한다.

#### 입력 항목
- `targetUserId`

#### 처리 규칙
- 요청자는 로그인한 사용자여야 한다.
- 자기 자신은 팔로우할 수 없다.
- 대상 사용자는 활성 회원이어야 한다.
- 이미 팔로우 중인 경우 중복 생성할 수 없다.

#### 데이터 반영
- `user_follow`에 `(follower, followee)` 레코드를 생성한다.
- `users.followee_count`, `users.follower_count`를 함께 갱신한다.

#### 예외
- 자기 자신 대상 요청이면 실패 처리한다.
- 이미 팔로우 중이면 중복 팔로우할 수 없다.
- 존재하지 않거나 탈퇴한 사용자는 팔로우할 수 없다.

### 5.5 사용자 언팔로우

#### 목적
- 로그인한 사용자가 기존 팔로우 관계를 해제한다.

#### 입력 항목
- `targetUserId`

#### 처리 규칙
- 기존 팔로우 관계가 있을 때만 언팔로우할 수 있다.
- 언팔로우 성공 시 즉시 관계 정보와 카운트가 반영되어야 한다.
- 이미 관계가 없는 경우 처리 방식은 API 정책으로 고정해야 한다. 기본값은 실패 처리로 정의한다.

#### 데이터 반영
- `user_follow`에서 `(follower, followee)` 레코드를 제거한다.
- `users.followee_count`, `users.follower_count`를 함께 갱신한다.

#### 예외
- 존재하지 않는 관계면 언팔로우할 수 없다.
- 존재하지 않거나 탈퇴한 사용자를 대상으로 한 요청은 실패 처리한다.

### 5.6 본인 활동 내역 조회

#### 목적
- 로그인한 사용자가 자신의 활동 내역 또는 활동 요약을 조회한다.

#### 입력 항목
- 조회 기간 또는 기준 월
- 페이지네이션 정보

#### 처리 규칙
- 활동 내역은 게시물 작성 기준 활동 단위로 구성한다.
- 월 단위 활동 여부를 반환한다.
- 삭제된 게시물, 댓글 기반 활동은 집계에서 제외한다.
- 미래 날짜는 허용하지 않는다.

#### 데이터 반영
- `posts`, `comments`, 필요 시 `group_members` 또는 그룹 활동 집계 데이터를 사용해 기간 기준으로 조회한다.

#### 예외
- 인증 정보가 없거나 유효하지 않으면 조회할 수 없다.
- 조회 기간 형식이 유효하지 않으면 실패 처리한다.
