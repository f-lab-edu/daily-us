# 피드 조회 API 구조 재설계

## 1. 문서 목적
- `docs/03_Test`의 테스트 결과를 바탕으로 현재 피드 조회 API 구조의 한계를 정리하고, 이를 개선하기 위한 재설계한다.

## 2. 현재 피드 조회 구조
- 현재 피드 조회 API는 `GET /api/v1/posts` 이다.
- 현재 구조는 사용자가 피드를 조회하는 시점에 필요한 데이터를 실시간으로 계산하는 Pull 방식
- 다음 순서로 동작한다.
  1. 사용자가 피드 대상이 있는지 확인한다.
  2. 사용자가 팔로우한 계정과 같은 그룹에 속한 사용자를 조회한다.
  3. 해당 사용자들의 게시물을 모은다.
  4. 게시물을 최신순으로 정렬한 뒤 필요한 개수만 반환한다.
  5. 반환 대상 게시물의 이미지 정보를 별도 조회한다.

## 3. 현재 구조의 문제점

### 3.1 사용자 관계 규모에 따라 조회 비용 증가
- 성능 테스트 결과, 일반 사용자는 빠르게 응답했지만 팔로우 수가 많은 사용자는 수 초가 소요
- `사용자가 맺고 있는 관계 수`에 따라 처리 비용이 높아짐

### 3.2 조회 시점의 계산량 과도
- 현재 구조에서는 최종적으로 10건만 반환, 하지만 먼저 대량의 게시물 후보군을 만든 뒤 정렬
- 팔로우 수가 많은 사용자는 관련 게시물 후보군이 증가
- 결과적으로 훨씬 많은 데이터를 읽고 정렬하기 때문에 비효율적

## 4. 재설계 방향
- 현재 구조의 핵심 문제는 `조회 시점에 너무 많은 계산을 수행`
- 따라서 조회 시점 계산을 줄인다.
- 이를 위해 피드 조회 구조를 fan-out 기반 구조로 재설계
- fan-out 구조에서는 게시물이 생성되는 시점에 사용자 피드를 계산하고, 사용자별 피드 저장소에 미리 반영
- 조회 시점에는 관계 집합을 계산하지 않고, 미리 적재된 사용자별 피드 목록을 조회
