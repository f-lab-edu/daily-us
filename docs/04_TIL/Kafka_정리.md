# Kafka 정리

## 1. Kafka란

Kafka는 대용량 이벤트를 빠르게 저장하고 전달하는 분산 메시징 플랫폼
단순한 메시지 큐보다, 이벤트 쌓아두고 여러 소비자가 독립적으로 읽을 수 있는 구조

## 2. 왜 쓰는가

- 요청 처리와 후속 작업을 분리할 수 있다.
- 하나의 이벤트를 여러 시스템이 함께 사용할 수 있다.
- 트래픽이 몰려도 이벤트를 버퍼링하며 처리할 수 있다.
- 장애가 나도 재처리하기 쉽다.

API 응답은 빠르게 끝내고 비동기로 넘길 때 유용하다.

## 3. 핵심 개념

### Producer
메시지를 Kafka로 보내는 주체

### Consumer
Kafka에서 메시지를 읽는 주체

### Topic
이벤트가 저장되는 논리적 채널

### Partition
Topic을 나눈 단위

- 여러 Partition으로 분산 저장 가능
- Consumer 병렬 처리 가능
- 같은 Partition 안에서만 순서 보장

### Offset
Partition 안에서 메시지의 위치
Consumer는 Offset으로 어디까지 읽었는지 관리

### Consumer Group
같은 Group의 Consumer들은 메시지를 나눠서 처리  
다른 Group이면 같은 메시지를 각자 독립적으로 소비할 수 있다.

## 4. Kafka의 장점

- 생산자와 소비자를 느슨하게 분리할 수 있다.
- 재처리와 확장이 쉽다.
- 로그성 데이터, 실시간 집계, 알림 처리에 강하다.

## 5. 전통적인 MQ와 차이

| 항목 | Kafka                                            | 전통적인 MQ                       |
| --- |--------------------------------------------------|-------------------------------|
| 기본 성격 | 이벤트 브로커                                          | 메시지 브로커                       |
| 데이터 저장 방식 | 메시지를 일정 기간 저장                                    | 메시지를 큐에 넣고 처리(Ack) 후 제거       |
| 소비 위치 관리 | Consumer가 Offset으로 직접 관리                         | Broker가 ack 기준으로 전달 상태 관리     |
| 재처리 | 데이터 내구성 중심                                       | 전달 완료 중심                      |
| 강한 영역 | 대용량 이벤트, 로그 보관, 재처리, 스트리밍 | 작업 분배, 요청-응답, 단기 비동기 처리 |

## 7. 주의할 점

- Kafka를 쓰면 운영 복잡도가 올라간다.
- 메시지 중복 처리 가능성을 고려해야 한다.
- 순서는 Topic 전체가 아니라 Partition 단위로만 보장된다.
